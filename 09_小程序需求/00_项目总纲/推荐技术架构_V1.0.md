# Bark & Bar 微信小程序｜推荐技术架构 V1.0

版本：V1.0｜日期：2026-07-09

## 1. 推荐结论

首店推荐：

- 顾客端：微信原生小程序 + TypeScript。
- 管理后台：Vue 3 + TypeScript + Vite + Element Plus。
- 后端：Node.js LTS + NestJS，模块化单体。
- 数据库：腾讯云 PostgreSQL。
- 缓存/锁：首期可不依赖 Redis；生产并发与任务锁需要时引入腾讯云 Redis。
- 文件：腾讯云 COS，私有桶 + 临时签名 URL。
- 部署：腾讯云轻量/CVM 或容器服务，Docker Compose 起步；数据库使用托管服务。
- 支付：微信支付 API v3。
- 监控：结构化日志、错误告警、慢查询、任务和对账告警。

理由：原生小程序对微信登录、手机号、支付、扫码和审核最稳；NestJS 适合权限、事务、模块和审计；PostgreSQL 约束、事务、JSONB 和报表能力适合财务/权益流水；模块化单体比微服务更适合首店团队与预算。

## 2. 方案比较

### 2.1 原生 vs Taro/uni-app

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| 微信原生 | 微信能力适配快、调试路径短、审核问题少 | 仅微信端复用 | 推荐 |
| Taro | React 团队复用、多端潜力 | 微信新能力有适配滞后 | 暂不选 |
| uni-app | 上手快、多端 | 复杂支付/扫码调试层更多 | 暂不选 |

首期只做微信生态，不应为未确定的多端需求支付复杂度。

### 2.2 云开发 vs 独立后端

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| 微信云开发 | 快速、运维轻 | 复杂事务、财务报表、可迁移性受限 | 原型可用，不作为正式核心 |
| 独立后端 | 数据模型、事务、审计和备份可控 | 需部署运维 | 推荐 |

### 2.3 MySQL vs PostgreSQL

两者均可。PostgreSQL 在约束、部分索引、JSONB、窗口函数和复杂分析上更匹配本项目，推荐 PostgreSQL 16+。团队若只有 MySQL 运维经验，可改用 MySQL 8，但必须保持相同事务和唯一约束。

### 2.4 Vue vs React

后台推荐 Vue 3，生态成熟、表单/表格开发效率高。若实施团队 React 更强，可选 React + Ant Design，不影响后端契约。

## 3. 逻辑架构

```text
微信小程序 ─┐
管理后台 ───┼─ HTTPS/API ─ NestJS 模块化单体 ─ PostgreSQL
老板数据中心 ┘                 │
                               ├─ 微信支付 API v3
                               ├─ COS 私有对象存储
                               ├─ Redis（按需）
                               └─ 定时任务/消息/监控
```

后端模块：

- Auth/RBAC、Users/Pets、Agreements
- Reservations/Admissions、Products/Cards/Benefits
- Orders/Payments/Refunds、Redemptions
- Catalog/Inventory/Procurement
- Campaigns/Coupons/Referrals
- Revenue/Deferred/Cash/Expenses
- Reports/Insights、Files、Audit、Jobs

## 4. 关键技术控制

### 4.1 微信登录

1. 小程序 `wx.login` 获取 code。
2. 服务端调用微信接口换取 `openid/session_key`。
3. 以 `app_id + openid` 唯一定位身份，签发短期 access token 和可轮换 refresh token。
4. 手机号通过微信手机号能力获取并加密保存。
5. 管理后台使用员工账号 + 强密码；老板/财务建议增加短信或 TOTP 二次验证。

### 4.2 微信支付

- 服务端创建业务订单和支付单，调用 JSAPI 下单。
- 回调验签、解密，使用微信交易号唯一约束。
- 回调与主动查单共用状态机和幂等键。
- 金额不符、订单关闭后付款等进入异常队列。
- 私钥、API v3 key 使用密钥管理，不入代码库。

### 4.3 防重复核销

- 动态码只包含随机 token/nonce，不暴露会员数据；有效期建议 30-60 秒（待确认）。
- Redis 或数据库保存 nonce 状态；首次消费后不可复用。
- 核销事务中锁定用户卡行，校验状态/有效期/本人/每日上限/余额。
- `idempotency_key`、`nonce`、业务核销号均唯一。
- 扣权益、写核销、写收入和写审计在同一数据库事务。

### 4.4 账务与库存

- 采用追加式业务流水，不提供删除接口。
- 汇总看板由流水计算；可做日汇总表提升速度，但明细是唯一事实源。
- 财务事件使用数据库事务 + outbox；异步任务可重试但不重复入账。
- 库存变动以流水计算，`inventory_balances` 是加速快照而非唯一事实。

## 5. 安全与隐私

- TLS 全链路；数据库/COS 私网访问优先。
- 手机号、证件号等字段应用层加密；列表接口脱敏。
- RBAC + 数据范围 + 敏感字段权限；拒绝默认开放。
- 所有管理写操作记录操作者、IP、设备、请求 ID、前后值。
- 上传限制类型/大小，私有桶、病毒/内容检查、短期签名访问。
- 防注入、限流、CSRF（后台）、XSS、重放和暴力登录。
- 按隐私政策设置保存期限和注销/删除流程；财务法定留存与个人删除请求需隔离处理。

## 6. 部署与备份

- 环境：local、test、staging、production；数据库、微信配置、支付证书和存储完全隔离。
- Docker 镜像固定版本；自动迁移必须先备份并支持回滚。
- PostgreSQL 托管自动备份 + 每日逻辑备份到独立存储；备份加密。
- 建议初始目标（待确认）：RPO 24 小时、RTO 4 小时；支付/订单流水额外提高备份频率。
- 每月恢复演练，记录恢复时间和抽样一致性。

## 7. Redis 与微服务判断

- Redis 不是首期硬依赖。单实例初期可用 PostgreSQL 唯一约束、行锁和任务表实现正确性。
- 当动态码、高并发限流、分布式任务/锁需要多实例时引入 Redis；Redis 只能加速，不能成为财务事实源。
- 不采用微服务。待多门店、独立团队或吞吐量证明模块化单体不足时再拆分。

## 9. 主要技术风险

- 微信主体认证、支付商户号和域名备案可能影响排期。
- 退款/收入规则未确认会阻塞财务模块最终实现。
- 动态码必须服务端实时校验，弱网需明确人工应急流程。
- 报表若直接依赖可编辑业务表而非流水，将导致历史数字漂移。
