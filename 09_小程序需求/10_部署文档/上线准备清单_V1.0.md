# Bark & Bar 微信小程序｜上线准备清单 V1.0

日期：2026-07-10｜适用：测试、预发、生产上线

## 1. 上线前资料

| 类别 | 必填项 | 状态 |
|---|---|---|
| 微信主体 | 小程序 AppID、主体认证、管理员微信 | 待补 |
| 支付 | 商户号、API v3 Key、商户证书、回调域名 | 待补 |
| 域名 | API域名、备案、HTTPS证书 | 待补 |
| 云资源 | 服务器、PostgreSQL、COS、日志、备份存储 | 待补 |
| 账号 | 老板、店长、财务、员工、后台管理员 | 待补 |
| 经营数据 | 商品、票卡、价格、库存、现金账户期初余额 | 待补 |
| 协议 | 服务协议、隐私政策、入园协议V1.4、会员/退款规则 | 待法律复核 |
| 会计税务 | 法定收入、应税收入、凭证期限、导出字段 | 待专业复核 |

## 2. 环境分层

| 环境 | 用途 | 要求 |
|---|---|---|
| local | 开发自测 | 可用模拟微信和支付回调 |
| test | 开发联调 | 数据可重置，不接真实支付 |
| staging | 上线彩排 | 尽量贴近生产，使用隔离商户/沙箱能力 |
| production | 正式经营 | 真实支付、真实用户、真实财务数据 |

生产和测试必须隔离：

- 数据库；
- 微信配置；
- 支付证书；
- COS存储桶；
- 环境变量；
- 管理员账号；
- 日志权限。

## 3. 密钥和配置

- 密钥不写入代码库。
- 生产密钥只允许老板授权的技术负责人和必要运维接触。
- 微信支付证书、API v3 Key、JWT密钥、数据库密码、COS密钥必须使用环境变量或密钥管理。
- 每次密钥变更记录变更人、时间、原因和影响范围。

## 4. 数据库上线步骤

1. 在测试库执行 `04_数据库设计/schema_v1.1.sql`。
2. 完成初始化数据：
   - 角色权限；
   - 商品分类；
   - 票卡模板；
   - 价格版本；
   - 协议版本；
   - 容量规则；
   - 现金账户；
   - 指标阈值。
3. 执行集成测试和数据勾稽。
4. 生产执行前做空库备份。
5. 生产执行DDL。
6. 导入初始化数据。
7. 抽样校验表数量、核心配置和管理员登录。

## 5. 备份与恢复

- RPO：24小时。
- RTO：4小时。
- PostgreSQL 托管自动备份。
- 每日逻辑备份到独立存储。
- 每周完整备份。
- 每月恢复演练一次，并记录：
  - 备份文件；
  - 恢复耗时；
  - 抽样订单/卡/库存/现金是否一致；
  - 发现问题和修复动作。

支付、退款、核销、现金流水建议额外保留更高频的审计日志和业务流水导出。

## 6. 发布检查

### 6.1 发布前

- 所有 P0 测试通过。
- OpenAPI 和后端实际接口一致。
- 数据库迁移可回滚或有备份恢复方案。
- 支付回调域名已配置并验证。
- 隐私政策和服务协议已在小程序内可见。
- 员工已完成签到、核销、弱网补录、退款申请、支出申请培训。
- 老板已确认首日可用容量、价格、账号权限和现金账户期初余额。

### 6.2 发布中

- 先发布后端，再发布后台，最后提交/发布小程序。
- 开启日志和错误告警。
- 首日安排技术值守，重点看：
  - 登录失败；
  - 支付回调；
  - 预约容量；
  - 核销重复；
  - 库存扣减；
  - 财务流水。

### 6.3 发布后

- 每日闭店核对收款、退款、核销、库存和现金账户。
- 试营业14天内，每日记录问题清单和修复状态。
- 重大规则修改必须新建版本并更新协议或业务说明。

## 7. 回滚和应急

| 问题 | 应急处理 |
|---|---|
| 小程序页面异常 | 暂停相关入口，前台改人工登记 |
| 支付异常 | 暂停线上支付，使用门店收款并登记补录 |
| 核销异常 | 停止动态码核销，店长记录待补录，老板复核 |
| 库存异常 | 暂停相关商品销售，盘点后冲正 |
| 退款异常 | 暂停自动执行，只保留申请和人工审核 |
| 数据库异常 | 按RTO方案恢复，恢复后核对支付/退款/核销流水 |

## 8. 生产不可妥协项

- 不允许真实离线核销。
- 不允许删除财务、库存、核销、协议、审计流水。
- 不允许普通员工获得全店财务权限。
- 不允许没有备份直接执行生产迁移。
- 不允许用测试商户号或测试密钥连接生产。
- 不允许协议和隐私政策未发布就正式收集用户资料。
