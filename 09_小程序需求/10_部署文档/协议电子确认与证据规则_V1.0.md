# Bark & Bar 微信小程序｜协议电子确认与证据规则 V1.0

日期：2026-07-10｜适用：入园须知及安全协议、用户服务协议、隐私政策、会员和退款规则

## 0. 当前协议来源

当前小程序《入园须知及安全协议》展示版应以 Bark&Bar 主项目协议为来源：

- 主项目源文件：`../02_入园协议/附件A_入园须知及安全协议_V1.3_统一序号版.docx`
- 小程序留档副本：`06_小程序前端/assets/agreements/entry-agreement-source-v1.3.docx`
- 小程序展示版本：V1.4

V1.4 是基于 V1.3 主协议整理的线上展示版，并同步后续已确认口径：泳池水深约0.8-1m、无浅水适应区、“宠物救生衣”表述，以及会员/退款按专项规则执行。不得把 V1.3 中与后续确认规则冲突的旧退款条款直接重新上线。

## 1. 结论

安全协议应作为小程序内可打开、可阅读、可追溯的完整协议正文，而不是只在页面上显示一句“我已阅读并同意”。

小程序上线时至少要做到：

1. 用户能在确认前打开完整协议；
2. 页面明确提示重点风险；
3. 用户主动勾选确认；
4. 系统保存签署证据；
5. 协议内容升级时要求重新确认；
6. 后台可按用户、时间、协议版本导出证据。

## 2. 小程序端展示方式

预约前必须展示：

- 预约前重点告知；
- 泳池风险；
- 犬只互动风险；
- 现场拒绝入园/入水权；
- 完整《入园须知及安全协议》入口；
- 明确勾选框和确认按钮。

不要只用一个弹窗让用户点“同意”。弹窗可以作为提示，但完整协议必须可查看。

## 3. 证据保存字段

正式后端应保存以下字段：

| 字段 | 说明 |
|---|---|
| user_id | 用户ID |
| openid/unionid | 微信身份标识 |
| phone_hash | 手机号哈希或脱敏值 |
| agreement_id | 协议ID |
| agreement_title | 协议标题 |
| agreement_version | 协议版本 |
| content_snapshot | 用户确认当时的协议全文快照 |
| content_hash | 协议全文哈希 |
| accepted_at | 确认时间 |
| ip | 请求IP |
| user_agent/device | 设备和客户端信息 |
| app_version | 小程序版本 |
| scene | 预约、购卡、入园等场景 |
| pet_ids | 关联宠物 |
| reservation_id/order_id/card_id | 关联业务单据 |
| checkbox_text | 用户勾选时看到的确认文字 |
| evidence_no | 签署证据编号 |

当前本地演示版已先保存：

- 协议标题；
- 协议版本；
- 确认时间；
- 内容快照；
- 内容哈希。

生产上线后需由后端保存，不应只存在手机本地。

## 4. 法律效力的关键点

从经营实务看，要提高线上确认的证明力，应做到：

- 能识别是谁点的同意；
- 能证明用户点同意前有机会阅读完整协议；
- 能证明用户同意的是哪个版本；
- 能证明协议内容确认后没有被悄悄改掉；
- 能在发生争议时导出完整证据链。

建议上线前让律师复核协议内容，尤其是免责、风险自担、退款扣除、个人信息和未成年人相关条款。协议不能排除依法不得排除的消费者权利或经营者责任。

## 5. 推荐交互

预约流程中：

1. 用户填写宠物档案；
2. 进入入园须知页；
3. 先看到三类重点风险；
4. 点击“查看完整《入园须知及安全协议》”；
5. 返回后勾选“我已阅读并同意……”；
6. 点击“确认并继续”；
7. 服务端保存签署证据；
8. 用户继续提交预约。

如果协议 V1.5 或更高版本发布：

- 未确认新版本的用户下次预约前必须重新确认；
- 旧版本签署记录不覆盖；
- 后台可查旧版本内容和确认记录。

## 6. 第三方电子签约

首期可以先采用小程序内电子确认 + 后端证据保存。

以下场景建议考虑第三方电子签约平台：

- 高金额会员卡；
- 重大事故后的专项确认；
- 股东/供应商/租赁等经营合同；
- 需要更强实名认证、数字证书、时间戳和司法存证能力的文件。

第三方平台不是所有入园用户都必须用，否则会显著增加操作成本。普通入园协议更适合在小程序内完成，但证据链必须做完整。
