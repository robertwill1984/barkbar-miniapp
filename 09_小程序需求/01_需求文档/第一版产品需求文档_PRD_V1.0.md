# Bark & Bar 微信小程序｜产品需求文档 PRD V1.0

版本：V1.0｜日期：2026-07-09｜状态：评审稿

## 1. 背景与目标

巴克巴需要一套用于真实门店的统一系统，替代分散的口头规则、聊天记录和表格。系统必须同时解决顾客体验、员工执行、会员权益、财务真实性和经营判断。

首期目标：

- 顾客可完成建档、预约、购票购卡、支付和查权益。
- 门店可完成健康/安全核验、签到、动态码核销、点单和库存处理。
- 老板可分别查看收款、收入、待兑付、退款、支出和现金。
- 所有关键操作可追溯、可对账、可冲正，不可静默删除。

非目标：首期不做多门店、配送、完整会计总账和复杂 AI 决策。

## 2. 用户与权限

| 角色 | 核心任务 | 敏感限制 |
|---|---|---|
| 顾客 | 预约、购票购卡、支付、查记录 | 仅本人及授权家庭数据 |
| 普通员工 | 接待、核验、核销、点单 | 不见现金余额/待兑付总额，不改财务规则 |
| 店长 | 日常运营、会员、库存、活动 | 财务字段按老板开关；关键调整需审批 |
| 财务/管理员 | 对账、支出、退款、月结 | 不可删除原记录 |
| 老板 | 全部经营管理和审批 | 高风险操作需二次确认 |

权限模型采用“角色权限 + 门店数据范围 + 敏感字段权限 + 操作审批”。

## 3. 核心用户故事与验收

### 3.1 登录与档案

顾客使用微信登录并绑定手机号，至少建立一个宠物档案后才能预约。

验收：

- `openid` 在同一小程序内唯一；手机号加密保存、后台默认脱敏。
- 宠物支持证件/疫苗图片、有效期和风险备注。
- 协议按版本留痕；新版本可要求再次确认。

### 3.2 预约

顾客选择日期、时段、人数、犬只和支付/卡项方式提交预约。

验收：

- 并发提交不突破人数、犬数和时段容量。
- 临时闭园、最晚预约/取消、审核和爽约规则可配置。
- 首次入园、健康信息过期、风险犬只进入人工审核。
- 预约修改保留版本；取消不删除原预约。

状态：`DRAFT → PENDING_PAYMENT/PENDING_REVIEW → CONFIRMED → CHECKED_IN → COMPLETED`；旁路为 `CANCELLED/NO_SHOW/REJECTED/EXPIRED`。

### 3.3 入园核验

员工按运营手册完成问询、疫苗/健康确认、5秒目测、绿黄红管控/禁入判断和协议核验。

验收：

- 禁入与红色管控分开；禁入不能通过普通员工放行。
- 首次或状态不确定犬只记录低密度适应。
- 预约签到与入园记录分开，必须保存人员/犬数和宠物。
- 泳池需求提示水深 0.8-1m、无浅水区；小狗或首次游泳必须确认游泳衣。

### 3.4 购票购卡与支付

顾客选择产品下单，系统创建预支付单并接收微信支付回调。

验收：

- 订单号、支付请求、支付回调、渠道交易号均幂等。
- 卡模板发布后，售出卡保存价格、有效期、权益和收入政策快照。
- 支付成功生成实际收款；不得直接把整笔卡款计入收入。
- 支付金额与订单应付不一致时冻结处理。

### 3.5 核销

顾客展示短时动态码；员工扫码并确认本次人数、犬数、宠物和扣次。

验收：

- 动态码短时有效、一次性 nonce、服务端签名校验，截图/重放失败。
- 10次卡本人必须到场、每日最多3次；季/年卡每日1次。
- 多狗可按规则追加扣次；超权益可生成加价订单。
- 核销成功原子化完成资格校验、次数扣减、入园关联和收入确认。
- 撤销只生成反向记录，需权限、原因和审批。

### 3.6 商城与库存

顾客点单并支付，员工制作/备货后交付。

验收：

- 订单状态：待支付、已支付、制作/备货中、待取、已完成、已取消、退款中、已退款。
- 交付后确认商品收入；销售出库与交付事件幂等关联。
- 库存不能直接改最终数，任何变化有流水。
- 库存不足时阻止超卖或进入明确的人工确认策略。

### 3.7 退款与售后

顾客申请，员工/财务审核，渠道退款后系统完成账务处理。

验收：

- 退款不超过可退余额；关联原订单和支付。
- 退款成功后冲减待兑付或已确认收入，并记录现金流。
- 老带新奖励和未使用优惠权益按规则撤销。
- 已关账期间通过调整记录处理。

### 3.8 经营数据

老板查看今日/月度指标并下钻。

验收：

- 每个指标展示定义、时间口径、更新时间、来源和明细。
- 收款、收入、待兑付、退款、支出、现金余额分开展示。
- 每日自动对账，差异明确列出。
- 建议必须包含触发阈值、计算式、样本期和相关明细。

## 4. 后台关键页面

- 登录与权限；今日工作台；预约日历；入园核验台；扫码核销台。
- 会员列表/详情；宠物风险档案；卡模板/用户卡详情。
- 商品/规格；订单；库存台账；采购/盘点；供应商。
- 活动/优惠券/老带新；支出；支付对账；退款审批。
- 收入确认；待兑付；现金流水；经营看板；报表中心。
- 员工/角色；规则配置；审计日志；备份/任务状态。

## 5. 业务规则

- 运营规则以 V1.7 手册和会员政策 V1.1 为基线。
- 卡项规则、财务政策和协议必须版本化。
- 金额以分存储；数量以整数最小单位；时间存 UTC、界面按 Asia/Shanghai。
- 所有财务、权益、库存采用追加流水；禁止物理删除。
- 人工调整必须保存前后值、原因、申请人、审批人和证据。

详细规则见：

- `11_经营规则/会员卡与收入确认规则_V1.0.md`
- `11_经营规则/现金流与运营支出管理规则_V1.0.md`

## 7. 非功能要求

- 核心支付/核销接口可用性目标 99.9%（待确认）。
- 支付与核销写操作必须幂等；关键事务满足原子性。
- 个人信息加密、最小权限、后台脱敏、访问日志。
- 数据库每日备份，至少每月恢复演练；RPO/RTO 待确认。
- 测试与生产微信环境、数据库、存储和密钥完全隔离。
- 页面支持弱网重试，但不得重复下单/核销。

## 8. 上线验收

- P0 全流程测试通过：预约、支付、核销、退款、支出、对账。
- 财务六账勾稽通过，随机抽样可追溯。
- 运营手册关键红线已编码并通过测试。
- 权限越权、重复回调、动态码重放、并发超卖测试通过。
- 备份恢复演练通过；门店员工完成培训与演练。

## 9. 依赖与待确认

- 微信小程序、支付商户号、主体认证、隐私指引。
- 腾讯云账号、域名、备案、SSL、短信/订阅消息。
- 退款折算、收入确认和税务口径。
- 预约容量、爽约、暂停/延期、审批阈值。
- 详见《待我确认的业务问题清单 V1.0》。
