# Bark & Bar 小程序后端服务

状态：MVP-A 本地数据库开发版｜日期：2026-07-13

## 定位

本目录是 Bark&Bar 主项目 `09_小程序需求/08_后端服务/` 下的后端代码骨架，不是独立项目。

首期目标不是一次性做完所有财务、支付、库存和退款，而是先打通正式上线必须依赖的基础闭环：

1. 微信登录占位；
2. 宠物档案；
3. 入园协议版本与签署证据；
4. 预约提交；
5. 审计、幂等、后续数据库迁移入口。

同时，后端必须预留财务分析能力，包括现金流、内部经营利润和股东分红依据。该部分先建立模块和公式边界，等支付、收入确认、支出和现金流水接入数据库后再形成真实金额。

## 当前技术选择

- Node.js 24+；
- NestJS 模块化单体；
- 本地开发阶段：SQLite 文件数据库；
- 正式服务器阶段：PostgreSQL 16+；
- 首期不强依赖 Redis；
- 正式微信登录、支付、COS 上传在后续接入真实账号后补齐。

## 当前本地数据库口径

- 当前后端已从“内存临时保存”改为“本地 SQLite 持久保存”。
- 默认数据库文件位置：`local-data/barkbar-local.sqlite`。
- `local-data/` 已加入 Git 忽略，不会把本地测试数据上传到 GitHub。
- 目前保留原接口结构：微信登录占位、宠物档案、协议确认、预约提交的核心逻辑不变。
- `InMemoryStore` 类名暂时保留，是为了减少本轮改动范围；实际底层已经写入 SQLite。

## 当前已建模块

- `src/modules/auth/`：微信登录占位、本地员工/老板 PIN 登录、本地 `staff_token` 过渡和会话校验；
- `src/modules/pets/`：宠物档案接口；
- `src/modules/agreements/`：协议版本、是否需确认、确认留证；
- `src/modules/reservations/`：预约时段、预约提交、预约查询、员工审核、员工 PIN、本地动态核验码、到店核验；
- `src/modules/cards/`：本地会员卡、会员 60 秒动态核销码、员工会员码核销、员工会员卡查询；
- `src/modules/orders/`：本地商品/票卡列表、订单创建、订单查询、订单支付状态查询、员工订单查询、本地模拟支付、现金流水和会员卡发放；
- `src/modules/refunds/`：本地退款/售后申请、单笔状态查询和员工审核，当前仅记录申请、通过、拒绝状态，不触发真实退款出账；
- `src/modules/finance/`：现金流、利润测算、股东分红依据，并为老板页提供订单、退款和会员卡汇总；本地开发阶段需老板 `staff_token` 或兼容 PIN 访问；
- `database/migrations/001_schema_v1.1.sql`：正式 PostgreSQL 阶段的数据库 V1.1 基线迁移。
- `src/common/store.service.ts`：本地 SQLite Store，支撑 MVP-A 本地接口联调。

## 本地员工/老板权限口径

当前仍是本地开发权限，不等同正式微信员工身份。后端已统一使用 `src/common/staff-auth.service.ts` 内的角色权限表校验后台接口：

| 权限 | STAFF | OWNER | 当前覆盖 |
| --- | --- | --- | --- |
| `reservation:review` | 是 | 是 | 预约列表、预约审核 |
| `reservation:checkin` | 是 | 是 | 到店核验、预约动态码核验 |
| `member:redeem` | 是 | 是 | 会员动态码核销 |
| `refund:review` | 是 | 是 | 退款申请列表、退款审核 |
| `member:read` | 是 | 是 | 会员卡只读查询 |
| `order:read` | 是 | 是 | 订单只读查询 |
| `finance:read` | 否 | 是 | 老板经营数据、利润测算、分红依据 |

正式上线时，这张权限表应替换为微信员工/老板身份、服务端 token 和数据库权限配置。

本地开发阶段已提供：

- `POST /admin/auth/pin-login`：用本地员工/老板 PIN 登录并返回 `staff_token`、角色和权限；
- `POST /admin/auth/session`：用 `staff_token` 或兼容 `staff_pin` 校验当前会话，并刷新角色和权限。

## 非程序员运行说明

你现在不用急着运行它。当前本地联调阶段只需要 Node.js 24+，暂时不需要正式服务器和微信真实 AppID。

1. 复制 `.env.example` 为 `.env`；
2. 保留 `BARKBAR_SQLITE_PATH=local-data/barkbar-local.sqlite`；
3. 安装依赖；
4. 启动本地服务；
5. 先用占位微信登录跑通宠物档案、协议确认、预约提交；
6. 再把小程序前端的本地假数据切换到真实 API。

当前 Codex 先完成代码骨架和规则对齐，不会替你生成真实微信密钥、支付密钥或生产数据库。

## 下一步开发顺序

1. 接入微信 `code2Session`；
2. 将本地员工/老板 PIN 登录替换为真实微信员工权限；

4. 根据 `11_经营规则/财务分析与股东分红规则_V1.0.md` 继续完善利润测算、待审退款预留、分红依据和正式老板权限；
5. 正式服务器确定后，再从 SQLite 开发版迁移到 PostgreSQL。
